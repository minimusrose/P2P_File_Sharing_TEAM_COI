🎯 OBJECTIFS DU PROJET
═══════════════════════════════════════════════════════════════════════════

Le projet vise à créer un système complet de partage de fichiers en 
peer-to-peer (P2P) permettant à plusieurs utilisateurs sur un réseau local 
de partager et télécharger des fichiers de manière décentralisée, sans 
serveur central.

Fonctionnalités Principales :
   ✅ Découverte automatique des peers sur le réseau (UDP broadcast)
   ✅ Partage de fichiers de toute taille
   ✅ Téléchargement avec fragmentation automatique (chunks de 256 KB)
   ✅ Vérification d'intégrité des fichiers (SHA256)
   ✅ Interface graphique intuitive (tkinter)
   ✅ Persistance des données (SQLite)
   ✅ Scripts d'installation et de déploiement
   ✅ Documentation complète utilisateur et développeur

2. PROTOCOLE RÉSEAU
────────────────────────────────────────────────────────────────────────

Le système utilise deux protocoles complémentaires :

A) UDP BROADCAST - Découverte des Peers
   Port : 5000
   Format : JSON
   Messages :
     • ANNOUNCE : Annonce périodique de présence (toutes les 30s)
     • GOODBYE : Déconnexion propre
   
   Avantages :
     ✓ Découverte automatique sans configuration
     ✓ Pas de serveur central requis
     ✓ Scalable pour réseaux locaux

B) TCP - Transfert de Données
   Ports : 5001+ (dynamique)
   Format : JSON pour métadonnées, bytes pour chunks
   Messages :
     • FILE_LIST_REQUEST  : Demande liste fichiers
     • FILE_LIST_RESPONSE : Envoi liste fichiers
     • CHUNK_REQUEST      : Demande d'un chunk spécifique
     • CHUNK_DATA         : Envoi données chunk
     • DOWNLOAD_COMPLETE  : Confirmation fin téléchargement
   
   Avantages :
     ✓ Connexion fiable avec vérification erreurs
     ✓ Ordre garanti des paquets
     ✓ Gestion sessions peer-to-peer


3. FRAGMENTATION ET INTÉGRITÉ
────────────────────────────────────────────────────────────────────────

Système de chunking intelligent :
  • Taille chunk : 256 KB
  • Hash : SHA256 par chunk + SHA256 fichier complet
  • Vérification : À chaque chunk + vérification finale
  
Processus de téléchargement :
  1. Requête métadonnées fichier
  2. Calcul nombre de chunks
  3. Téléchargement séquentiel des chunks
  4. Vérification hash de chaque chunk
  5. Assemblage dans fichier final
  6. Vérification hash fichier complet

Avantages :
  ✓ Détection corruption immédiate
  ✓ Possibilité de reprise (future)
  ✓ Téléchargement parallèle depuis plusieurs peers (futur)

💻 CHOIX TECHNIQUES ET JUSTIFICATIONS

1. LANGAGE : PYTHON 3.9+
────────────────────────────────────────────────────────────────────────

Justification :
  ✓ Multiplateforme (Windows, Linux, macOS)
  ✓ Bibliothèque standard complète (socket, tkinter, sqlite3)
  ✓ Pas de dépendances externes requises
  ✓ Syntaxe claire et maintenable
  ✓ Excellent pour prototypage rapide
  ✓ Threading natif pour concurrence

Bibliothèques utilisées (stdlib uniquement) :
  • socket       - Communication réseau UDP/TCP
  • tkinter      - Interface graphique
  • sqlite3      - Base de données
  • threading    - Concurrence et parallélisme
  • hashlib      - Hachage SHA256
  • json         - Sérialisation messages
  • pathlib      - Manipulation chemins cross-platform
  • logging      - Journalisation


2. INTERFACE GRAPHIQUE : TKINTER
────────────────────────────────────────────────────────────────────────

Justification :
  ✓ Inclus dans Python (pas d'installation externe)
  ✓ Multiplateforme natif
  ✓ Léger et performant
  ✓ Suffisant pour interface simple et fonctionnelle
  ✓ Large documentation disponible

Composants développés :
  • Liste peers connectés avec auto-refresh
  • Liste fichiers disponibles avec tri automatique
  • Barre de progression téléchargement
  • Boutons d'action (partager, télécharger, supprimer)
  • Colonne "Disponibilité" (Local/Distant/Non disponible)
  • Sélection intuitive par double-clic


3. BASE DE DONNÉES : SQLITE
────────────────────────────────────────────────────────────────────────

Justification :
  ✓ Embarqué (pas de serveur séparé)
  ✓ Fichier unique portable
  ✓ Thread-safe avec contrôle approprié
  ✓ SQL standard (requêtes complexes possibles)
  ✓ Transactions ACID (intégrité garantie)
  ✓ Léger (~600 KB binaire)

Alternatives écartées :
  ✗ JSON - Pas de requêtes, corruption facile
  ✗ MySQL/PostgreSQL - Overkill, configuration complexe
  ✗ NoSQL - Pas adapté aux relations peers/files


4. ARCHITECTURE RÉSEAU : UDP + TCP
────────────────────────────────────────────────────────────────────────

Justification UDP pour découverte :
  ✓ Broadcast efficace sur réseau local
  ✓ Pas de handshake (overhead minimal)
  ✓ Parfait pour annonces périodiques
  ✓ Découverte automatique sans configuration

Justification TCP pour transferts :
  ✓ Fiabilité critique pour données fichiers
  ✓ Ordre garanti des chunks
  ✓ Contrôle de flux intégré
  ✓ Détection d'erreurs automatique
  ✓ Sessions peer-to-peer persistantes

Alternative écartée :
  ✗ WebSockets - Dépendance externe, complexité inutile
  ✗ UDP uniquement - Pas fiable pour gros fichiers


5. GESTION CONCURRENCE : THREADING
────────────────────────────────────────────────────────────────────────

Justification :
  ✓ Natif Python (pas de dépendance)
  ✓ Suffisant pour I/O-bound (réseau, fichiers)
  ✓ Simple à implémenter
  ✓ Compatible tkinter (avec queue/after)

Threads utilisés :
  • Thread broadcast UDP (envoi périodique)
  • Thread listen UDP (réception annonces)
  • Thread par connexion TCP
  • Thread mise à jour GUI
  • Thread téléchargement fichier

Alternative écartée :
  ✗ asyncio - Complexité inutile pour ce projet
  ✗ multiprocessing - Overhead mémoire, IPC complexe


6. PACKAGING : SCRIPTS BATCH/SHELL
────────────────────────────────────────────────────────────────────────

Justification :
  ✓ Pas de compilation nécessaire
  ✓ Scripts simples et auditables
  ✓ Installation vérifiée (Python, tkinter)
  ✓ Création automatique dossiers
  ✓ Messages d'erreur clairs

Scripts créés :
  • install.bat/sh    - Vérification et initialisation
  • run.bat/sh        - Lancement application
  • run_debug.bat     - Mode debug (logs visibles)
  • uninstall.bat/sh  - Nettoyage données

Alternative écartée :
  ✗ PyInstaller - Binaires volumineux, bugs cross-platform
  ✗ cx_Freeze - Complexité configuration
  ✗ Docker - Overkill pour application desktop

🎯 DIFFICULTÉS RENCONTRÉES ET SOLUTIONS
═══════════════════════════════════════════════════════════════════════════

1. DÉCOUVERTE UDP INITIALE
────────────────────────────────────────────────────────────────────────
Problème : Les peers ne se découvraient pas toujours immédiatement
Solution : 
  • Ajout d'un système de retry automatique
  • Augmentation fréquence broadcast initial
  • Ajout logs debug détaillés
  • Documentation diagnostic (DIAGNOSTIC_DECOUVERTE.md)

2. SYNCHRONISATION GUI ET THREADS
────────────────────────────────────────────────────────────────────────
Problème : Mises à jour GUI depuis threads provoquaient des erreurs
Solution :
  • Utilisation de root.after() pour updates GUI safe
  • Queue pour communication thread → GUI
  • Callbacks asynchrones pour événements réseau

3. GESTION DÉCONNEXION PEERS
────────────────────────────────────────────────────────────────────────
Problème : Peers déconnectés restaient marqués comme "disponibles"
Solution :
  • Système de ping/pong périodique
  • Timeout de 2 minutes pour marquer peer inactif
  • Mise à jour automatique colonne "Disponibilité"
  • Nettoyage base de données régulier

4. INTÉGRITÉ DES CHUNKS
────────────────────────────────────────────────────────────────────────
Problème : Corruption occasionnelle lors de transferts volumineux
Solution :
  • Vérification hash SHA256 de chaque chunk
  • Retry automatique en cas d'échec
  • Vérification hash fichier complet à la fin
  • Logs détaillés pour traçabilité

5. PORTS DÉJÀ UTILISÉS
────────────────────────────────────────────────────────────────────────
Problème : Application plantait si port 5001 déjà utilisé
Solution :
  • Détection automatique port occupé
  • Incrémentation port jusqu'à trouver port libre
  • Annonce du port TCP dynamique dans broadcast UDP
  • Message d'erreur clair si aucun port disponible
